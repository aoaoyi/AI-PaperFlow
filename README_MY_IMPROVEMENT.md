# My Improvements to Paper Daily

## Project Background

This project is based on the open-source project [Futuresxy/paper-daily](https://github.com/Futuresxy/paper-daily). I reproduced the original workflow and made several secondary development improvements on top of it.

The original project is designed to automatically track new arXiv papers, evaluate their relevance to configured research interests, generate Chinese summaries, and publish a static paper-reading page through GitHub Actions and GitHub Pages.

## My Improvements

My secondary development work focuses on improving paper ranking quality, reducing duplicated entries, and making the ranking result easier to understand in the web UI.

The main improvements include:

- Personalized `final_score` ranking
- Paper deduplication
- RAG corpus export
- Qwen/OpenAI-compatible RAG answer generation
- Optional ChromaDB semantic retrieval
- Cohere reranking fallback
- Gemini-based RAG answer generation
- DeepResearch Lite report generation
- Daily email digest
- Evaluation metrics for retrieval quality and latency
- Final score display in paper cards
- Unit tests for ranking and deduplication logic

## Personalized Ranking Design

The original project keeps a relevance score for each paper. I kept this original score unchanged and added a new `final_score` field for personalized ranking.

The ranking idea is:

```text
final_score = original score + keyword_bonus + freshness_bonus + pdf_bonus
```

In the implementation, the final score combines several factors:

- `original score`: the original relevance score generated by the project, kept as the base relevance signal.
- `keyword_bonus`: calculated from keyword hits in the paper title and abstract.
- `freshness_bonus`: calculated from the paper publication date, so newer papers can be ranked higher when relevance is similar.
- `pdf_bonus`: used to slightly prioritize papers with available PDF links.

This design keeps the original relevance score available for inspection, while using `final_score` as a more practical ranking score for daily reading.

## Paper Deduplication Design

I added a deduplication step before papers enter the final displayed JSON data.

The deduplication rules are:

- Papers with the same arXiv ID are treated as duplicates.
- If no arXiv ID is available, papers with the same DOI are treated as duplicates.
- If neither arXiv ID nor DOI is available, a normalized paper title is used for deduplication.
- When duplicated papers are found, the system keeps the more complete record.

Title normalization converts titles to lowercase and removes extra spaces and common punctuation. For duplicated records, completeness is judged by metadata such as PDF link availability, abstract availability, Chinese summary availability, authors, and categories.

## Frontend Display

I updated the frontend paper cards to display `final_score` when it is available.

The original HIGH / MEDIUM / LOW relevance score badge is still preserved. The additional final score display makes the personalized ranking result more transparent to users.

I also added a separate local demo page for AI-PaperFlow research workflows. This page does not replace the original GitHub Pages paper display page. It provides Paper QA, DeepResearch Lite report generation, retrieved paper cards, and browser-based reading status management.

## RAG Corpus and QA

I added a standalone RAG corpus export step that converts the existing paper JSON files into `web/data/rag_corpus.json`. This keeps the original display data unchanged while providing a normalized format for retrieval and question answering.

The local RAG API supports:

- TF-IDF retrieval as the default fallback.
- Gemini answer generation through environment variables.
- Optional Qwen/OpenAI-compatible embeddings with ChromaDB semantic retrieval.
- Optional Cohere reranking with fallback to simple score fusion.

## DeepResearch Lite

The project now includes a simplified research agent workflow:

- Planner: decomposes a research topic into sub-questions.
- Retriever: retrieves papers for each sub-question.
- Evidence Aggregator: compresses retrieved paper metadata into an evidence pack.
- Writer: generates a structured Chinese report when the LLM is available.
- Verifier: checks whether cited paper titles are supported by retrieved papers.

If the LLM, ChromaDB, embedding API, or reranker is unavailable, the workflow falls back to rule-based planning, BM25/lexical retrieval, and retrieval-only summaries.

## Daily Email Digest and Evaluation

I added a daily digest script that selects the top papers by `final_score` and sends them through SMTP when configured. If SMTP configuration is missing, the digest is printed to logs instead of failing.

I also added a lightweight retrieval evaluation script that measures Top-1/Top-3 hit rate and average latency for TF-IDF retrieval, and optionally Milvus retrieval when available.

## Testing

I added unit tests in `tests/test_collect_papers.py` to verify the new logic.

The tests cover:

- `final_score` calculation
- arXiv ID based deduplication
- DOI based deduplication
- normalized title based deduplication
- keeping the more complete paper record when duplicates are detected

Additional validation scripts include:

- RAG corpus export validation
- Python compile checks for backend modules
- Retrieval evaluation through `eval_paperflow.py`

## Result

The project can still run through the original GitHub Actions workflow and deploy the generated static website to GitHub Pages.

These improvements do not change the original deployment architecture. They extend the data processing and frontend display behavior while keeping the original project workflow intact.
